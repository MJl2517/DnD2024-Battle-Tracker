# Архитектура DnD 2024 Battle Tracker

## Принципы

- Функции приложения организованы вертикальными модулями: кампании, игроки, бестиарий, энкаунтеры, бой, экран игроков и настройки.
- Renderer работает только через типизированный `window.dndTracker`. Доступа к Node.js, SQLite и строковым IPC-каналам в React-коде нет.
- Main-процесс координирует окна, IPC и application services. SQL остаётся внутри репозиториев.
- Shared-код не зависит от Electron или React и содержит доменные типы, контракты IPC и чистую бизнес-логику.
- Внешние сайты подключаются отдельными адаптерами. Ошибка одного источника не должна менять правила разбора другого.

## Поток данных

```text
React feature -> TrackerApi -> preload -> typed IPC -> handler -> service -> repository -> SQLite
                                                       |
                                                       +-> public player view event
```

Главное окно является источником управляющих действий. Публичное окно получает только подготовленное `PublicCombatView` и не изменяет бой.

## Renderer

Каждая функция размещается в `src/renderer/src/features/<feature>`. Компоненты, используемые несколькими функциями, находятся в `shared/ui`. Чистые преобразования данных не должны жить внутри JSX-компонентов.

Локальное состояние хранится в hooks. Context или `useReducer` используются только когда состояние действительно разделяется несколькими компонентами одной функции. Глобальная state-библиотека не требуется.

## Main и хранение

Точка входа собирает приложение из модулей окон, обновления и IPC. `TrackerRepository` сохраняется как совместимый фасад, а специализированные репозитории владеют запросами своих агрегатов. Транзакции должны охватывать целиком пользовательскую операцию, а не отдельный SQL-запрос.

Схема SQLite изменяется только через идемпотентную миграцию и проверяется тестом на копии старой базы.

## Комментарии

Комментарии объясняют причины, инварианты, ограничения Electron/SQLite и особенности внешних источников. Очевидные присваивания и JSX не комментируются. Публичные контракты и расширяемые адаптеры документируются через JSDoc.
